# Data Model: User Authentication System

**Feature**: `001-user-auth` | **Date**: 2026-05-12

---

## Entities

### User

Represents a registered portal employee.

| Column | Type | Constraints | Notes |
|--------|------|-------------|-------|
| `id` | `TEXT` | `PRIMARY KEY` | UUID v4, generated by application |
| `full_name` | `TEXT` | `NOT NULL` | Free text, non-empty (FR-013) |
| `email` | `TEXT` | `NOT NULL UNIQUE` | Normalised to lowercase before storage; validated as RFC 5322 email (FR-013) |
| `hashed_password` | `TEXT` | `NOT NULL` | bcrypt hash via passlib (R-002); plaintext never stored |
| `role` | `TEXT` | `NOT NULL CHECK(role IN ('admin','submitter'))` | Assigned at registration; only `submitter→admin` promotion allowed (ADR-006 D7) |
| `privacy_policy_accepted` | `INTEGER` | `NOT NULL DEFAULT 0` | `1` = accepted; registration blocked if `0` (FR-002) |
| `created_at` | `TEXT` | `NOT NULL DEFAULT (datetime('now'))` | ISO 8601 UTC timestamp |

**Uniqueness rule**: `email` is globally unique across all users. Registration with a
duplicate email returns HTTP 409 (FR-003).

**State transitions — role**:
```
submitter ──promote──► admin
```
Admin → Submitter demotion is out of scope (ADR-006 D7). Role is set once at registration
(first user = `admin`, all others = `submitter`; ADR-006 D6) and may only change upward.

### Session

Represents one active authenticated session for a user.

| Column | Type | Constraints | Notes |
|--------|------|-------------|-------|
| `id` | `TEXT` | `PRIMARY KEY` | UUID v4, generated by application |
| `user_id` | `TEXT` | `NOT NULL REFERENCES users(id)` | FK to User; not cascade-deleted (user deletion out of scope) |
| `token` | `TEXT` | `NOT NULL UNIQUE` | Full JWT string as delivered in the httpOnly cookie |
| `expiry_time` | `TEXT` | `NOT NULL` | ISO 8601 UTC; 8 hours after creation (ADR-006 D1) |
| `created_at` | `TEXT` | `NOT NULL DEFAULT (datetime('now'))` | ISO 8601 UTC timestamp |

**Lifecycle**:
```
Login ──► INSERT session row + set httpOnly cookie
Logout ──► DELETE session row (matched by token) + clear cookie   [FR-008, ADR-006 D1]
Expiry ──► row detected stale by middleware (expiry_time < now)   [ADR-006 D5]
           → DELETE row + raise 401 → frontend redirects to login [SC-006]
```

**Concurrent sessions**: Multiple rows per `user_id` are permitted (ADR-006 D2).
Each login creates a new row. Logout deletes only the row matching the cookie token.

## DDL

```sql
-- Users table
CREATE TABLE IF NOT EXISTS users (
    id                     TEXT NOT NULL PRIMARY KEY,
    full_name              TEXT NOT NULL,
    email                  TEXT NOT NULL UNIQUE,
    hashed_password        TEXT NOT NULL,
    role                   TEXT NOT NULL CHECK(role IN ('admin', 'submitter')),
    privacy_policy_accepted INTEGER NOT NULL DEFAULT 0,
    created_at             TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now'))
);

CREATE INDEX IF NOT EXISTS idx_users_email ON users(email);

-- Sessions table
CREATE TABLE IF NOT EXISTS sessions (
    id          TEXT NOT NULL PRIMARY KEY,
    user_id     TEXT NOT NULL REFERENCES users(id),
    token       TEXT NOT NULL UNIQUE,
    expiry_time TEXT NOT NULL,
    created_at  TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now'))
);

CREATE INDEX IF NOT EXISTS idx_sessions_token      ON sessions(token);
CREATE INDEX IF NOT EXISTS idx_sessions_expiry     ON sessions(expiry_time);
CREATE INDEX IF NOT EXISTS idx_sessions_user_id    ON sessions(user_id);
```

## Query Analysis (Principle II)

All hot-path queries use indexed columns. No N+1 patterns.

| Query | Columns Used | Index | Notes |
|-------|-------------|-------|-------|
| Login — find user by email | `users.email` | `idx_users_email` | Single row lookup |
| Auth middleware — validate session | `sessions.token` | `idx_sessions_token` | Single row PK-equivalent lookup |
| Lazy expiry check | `sessions.expiry_time` | checked per-row after token lookup | Combined in one query: `WHERE token = ? AND expiry_time > ?` |
| First-user count | `COUNT(*) FROM users` | full scan, 0 rows initially | Only during registration; acceptable |
| User list (Admin) | `SELECT * FROM users` | full scan | Internal tool; expected row count ≤ hundreds; no pagination needed for MVP |
| Promote user | `users.id` | `PRIMARY KEY` | Single row update by PK |

## Validation Rules

Enforced by Pydantic schemas at the API boundary (FR-013):

| Field | Rule |
|-------|------|
| `full_name` | Non-empty string; min length 1 after stripping whitespace |
| `email` | Valid email format (Pydantic `EmailStr`); normalised to lowercase |
| `password` | Minimum 8 characters (spec Assumptions); no maximum enforced |
| `privacy_policy_accepted` | Must be `true`; registration rejected if `false` (FR-002) |
