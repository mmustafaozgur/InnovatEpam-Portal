# Implementation Plan: Forgot Password & My Profile Page

**Branch**: `008-forgot-password-profile` | **Date**: 2026-05-14 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/008-forgot-password-profile/spec.md`

---

## Summary

Add two self-service features to the InnovatEpam Portal:

1. **Forgot Password** — an inline reset form on the login page that directly updates the stored password (no email, no token, no page navigation). Enabled by a new `POST /api/v1/auth/reset-password` endpoint.

2. **My Profile** — a protected `/profile` page with two vertical card sections: Account Information (editable full name, read-only email) and Change Password (requires current password). Accessible to all authenticated users via a new Sidebar link. Backed by two new endpoints: `PATCH /api/v1/users/me` and `POST /api/v1/users/me/change-password`.

---

## Technical Context

**Language/Version**: Python 3.11 (backend) · TypeScript 5.x (frontend)

**Primary Dependencies**: FastAPI · Pydantic · SQLAlchemy async · passlib/bcrypt (backend); React + Vite · react-hook-form · Zod · shadcn/ui · Tailwind CSS (frontend)

**Storage**: SQLite via SQLAlchemy async — **no schema migration required** (User table already has `full_name`, `email`, `hashed_password`; no new tables)

**Testing**: pytest + pytest-asyncio + httpx AsyncClient (backend) · Vitest + React Testing Library (frontend)

**Target Platform**: Local/intranet web application

**Project Type**: Full-stack web application (FastAPI backend + React frontend)

**Performance Goals**: sub-100ms p95 on all new endpoints — all queries are single-row SELECT on indexed `email` or `id` columns plus one UPDATE; no N+1 queries possible

**Constraints**: No email infrastructure. Session must NOT be invalidated after password change via My Profile. Inline forgot-password with enumeration disclosure is intentional per ADR-013.

**Scale/Scope**: Single-tenant intranet; tens to low-hundreds of users

## Implementation Notes

### Backend — key implementation details

**`auth_service.reset_password(db, data)`**:
- Normalize email to lowercase
- `SELECT` user by email; raise `404` if not found (per FR-007 — specific error, not generic)
- `hash_password(data.new_password)` and assign to `user.hashed_password`
- `await db.commit()`

**`user_service.update_profile(db, user_id, full_name)`**:
- `SELECT` user by `id` (PK, always found since caller is authenticated)
- Assign `user.full_name = full_name.strip()`; commit; refresh; return user

**`user_service.change_password(db, user_id, current_password, new_password)`**:
- `SELECT` user by `id`
- `verify_password(current_password, user.hashed_password)` → `400` if wrong
- `user.hashed_password = hash_password(new_password)`; commit
- Session token unchanged (no session table touch)

### Frontend — key implementation details

**`AuthContext.tsx`**:
- Add `updateUser: (user: User) => void` to `AuthContextValue`
- Implement as `const updateUser = useCallback((u: User) => setUser(u), [])`
- Expose in provider value

**`LoginForm.tsx`**:
- Add `const [showForgot, setShowForgot] = useState(false)`
- Below the form, render `<button onClick={() => setShowForgot(v => !v)}>Forgot password?</button>`
- Conditionally render `<ForgotPasswordForm onSuccess={() => setShowForgot(false)} />`

**`ForgotPasswordForm.tsx`**:
- Three fields: email, new_password, confirm_password
- Zod `forgotPasswordSchema` with `refine` for password match
- On success: show success alert, call `props.onSuccess()` to hide form
- On 404: show "No account found with that email address."

**`Sidebar.tsx`**:
- Add to `NAV_ITEMS`: `{ to: '/profile', label: 'My Profile', icon: User, roles: ['submitter', 'admin'] }`
- Import `User` from `lucide-react` (already imported in other files)

**`App.tsx`**:
- Add lazy import for `ProfilePage`
- Inside `<Route element={<ProtectedRoute />}> <Route element={<AppLayout />}>`, add `<Route path="/profile" element={<ProfilePage />} />`

**`ProfilePage.tsx`**:
- Renders two `<section>` / card elements: `<AccountInfoSection />` and `<ChangePasswordSection />`
- Page title: "My Profile"
- Layout: single-column, vertically stacked cards with `space-y-6`

**`AccountInfoSection.tsx`**:
- Uses `useAuth()` to get `user` and `updateUser`
- Pre-fills `full_name` from `user.full_name`; email field is `disabled`
- On save: `updateProfile({ full_name })` → on success → `updateUser(responseUser)` + success alert
- Button disabled while submitting (prevents double-click per FR — edge case)

**`ChangePasswordSection.tsx`**:
- Three password fields: current_password, new_password, confirm_password
- On submit: `changePassword({ current_password, new_password })`
- On success: reset form + show success alert
- On 400: show "Current password is incorrect."
- Button disabled while submitting

**Shared validation rules across Zod schemas**: `forgotPasswordSchema`, `accountInfoSchema`, and `changePasswordSchema` each independently implement the same two rules — minimum 8-character password and password-match refinement. This is intentional per Principle V (no premature abstraction — three schemas does not justify a shared helper). If these rules ever need to diverge in behavior across forms, that divergence must be a conscious, documented decision at the point of change, not silent drift.
